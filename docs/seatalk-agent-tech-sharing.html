01 — OVERVIEW
这是什么项目？
一个注入到 SeaTalk 桌面端里的 AI 工作台，不是聊天玩具，而是面向数据研发排障和开发的工作入口
🧩
项目定位
SeaTalk Agent 是 SPX Helper 的一个核心模块。它把 AI 面板直接注入 SeaTalk，让用户在聊天窗口里完成问答、排障、查数据、远程控制和自动回复。
👤
谁在用
主要面向 Shopee 数据工程师和运维排障场景，尤其适合“告警先到群里、上下文先在消息里”的日常工作流。
🛠️
它能做什么
读当前聊天、引用消息、调用 MCP 查 Presto / CK / Flink / Scheduler / API Trace、触发告警自动排查、执行 Remote 指令，还能把回复直接发回 SeaTalk。
🏗️
为什么值得讲
这个项目不是单点脚本，而是一套“宿主注入 + Agent Runtime + MCP 工具 + Skill 工作流”的框架，所以后面讨论的不是某个小功能，而是一整套可迁移的架构。
可以把它理解成：把一个能读代码、会查数据、能调工具的工程 Agent，嵌进公司最常用的聊天软件里。

05 — BRIDGE
双向通信桥
注入的前端代码如何与 Node 后端实时通信？
页面 → Node：Runtime.addBinding
CDP 提供 Runtime.addBinding('__agentSend')，在页面注入一个全局函数。

页面调用 window.__agentSend(json) 时，CDP 触发 Runtime.bindingCalled 事件，Node 端收到数据。

特点：原生 Chromium 机制，不走网络，零延迟。
Node → 页面：Runtime.evaluate
Node 端调用 Runtime.evaluate，在页面执行任意 JS 代码。

统一通过 window.__agentReceive(json) 将数据推送到前端。

优化：文本流式传输做了 80ms 批量合并，减少 evaluate 频率。
// Node → 页面（推送 AI 回复） await client.evaluate( `window.__agentReceive && window.__agentReceive(${JSON.stringify(msg)})` ); // 页面 → Node（用户发送消息） // 页面内调用： window.__agentSend(JSON.stringify({ type: 'user_message', text: '帮我查一下...' })); // Node 端自动收到 Runtime.bindingCalled 事件

09 — MIGRATION
为什么迁移到 Codex？
不是推倒重来，而是在保留原有技术选型的前提下，替换 Agent 后端并整理 Cursor 时代资产
迁移时保留了什么
Inspector + CDP Proxy、Runtime.addBinding + Runtime.evaluate、四个注入脚本、MCP servers、Alarm Bot、Remote 控制都继续保留。

也就是说，SeaTalk Agent 的宿主层、桥接层和数据工具层没有因为迁移而被重写，变化主要集中在 Agent 接入层。
迁移的理由
• 直接原因很现实：公司不再允许继续使用 Cursor，所以 Agent 后端必须迁走
• 这次迁移证明了框架兼容性强：只替换 Agent Runtime，SeaTalk 注入、Bridge、MCP、Skill、Alarm Bot、Remote 控制都可以继续复用
• 兼容层足够薄：前端 UI 和宿主侧只关心“有一个可对话、可调工具的 Agent”，不把实现细节绑死在单一厂商协议上
• 结果就是可迁移：从 Cursor ACP 切到 Codex App Server，不需要推翻整套架构
迁移过程
1. 从 .cursorrules 提炼长期稳定规则，落到 AGENTS.md
2. 从 ~/.cursor/mcp.json 恢复旧 MCP，拆成仓库模板 .codex/config.toml 与本机真实配置
3. 用 scripts/codex-mcp-launch.sh 收敛 MCP 启动方式、路径定位与 PYTHONPATH
4. 把高价值 Cursor skills 迁成 .agents/skills/ 的瘦入口，并按需挂 references
5. 在 SeaTalk Agent 中加入 codex-app-server，同时保留 cursor-acp 作为兼容回退
迁移时踩住的边界
• 私有 token、绝对路径、service account 继续留在本机 ~/.codex/config.toml
• .cursor/skills/ 保留为 Cursor 时代原始知识库，不假装 Codex 会自动继承
• 验证不只看“能启动”，还要做真实 tools/call、skill 可见性和实际工作流验证
对应仓库文档：docs/guides/CURSOR_TO_CODEX_MIGRATION.md、docs/guides/CODEX.md、docs/guides/SEATALK_AGENT.md

12 — ALARM BOT
群内告警自动排查
匹配规则后拉起 AI 排查，结果以线程回复发回群聊
流程
侧栏腕表入口配置监控群与匹配规则（正则，支持与/或）、处理延迟、工作区与追加 prompt。群内出现告警且满足条件时，后端 investigateAlarm 创建 Agent 会话执行排查（可搭配 Flink MCP、flink-alert-triage Skill），完成后对告警消息做线程回复。
消息外层与 prompt 约定
发送前 Agent 会统一加上标题 [Alarm Bot] Auto-Investigation 与上下分隔线。因此追加 prompt 与 Skill 模板应要求模型只输出正文（任务名、严重度、结论、链接等），不要重复写外层标题或整行装饰分隔线，避免双层排版。发送前亦可对模型输出做去重剥离（与规范二选一或叠加）。
配置与安全
配置持久化：~/.cursor/seatalk-alarm-config.json

支持授权发送者白名单、仅 @ 我、人工回复取消自动排查等开关，降低误触发与 prompt 注入面。
详细说明见仓库 docs/guides/SEATALK_AGENT.md 中「Alarm Bot」章节与 .cursor/skills/flink-alert-triage/。
13 — REMOTE CONTROL
Remote 远程控制
用手机给 AI 下指令——通过 SeaTalk Saved Messages 远程操控 SeaTalk Agent
工作原理
01
消息监听
seatalk-watch.js 通过 Redux store.subscribe 实时监听所有消息变更
02
识别指令
筛选 buddy-{myUid}（Saved Messages）的消息，识别为远程指令
03
分发处理
!! 前缀 → 系统指令（直接响应）
普通文本 → 独立 Remote Agent 进程执行
04
回复
结果以 Markdown 卡片格式发回 Saved Messages，手机端清晰区分指令与回复
!! 系统指令（不走 AI）
!!ping — 存活检测
!!status — 运行状态
!!help — 指令列表
!!use <model> — 远程切换模型
!!ls models — 列出可用模型
!!reset — 重置 Agent 会话
!!remote on/off — 远程开关控制
消息冻结机制
连续发两条指令？第一条自动冻结（跳过回复），只执行最新一条。

remoteFrozen 标记当前消息被覆盖
remoteMessageQueue 缓冲新到达的指令
finally 块 取出排队指令继续处理
防循环机制
前端 _sentTexts + "Remote Agent" 前缀过滤 → 后端 startsWith 二次拦截 → 杜绝自身回复被当作新指令
14 — AI-DRIVEN DEV
所见即所得的 AI 开发模式
用自然语言描述需求，AI 写代码、你验证效果、迭代优化——整个过程就是开发
开发理念转变
  • 传统模式：阅读文档 → 理解 API → 手写代码 → 调试 → 迭代
  • AI 模式：描述意图 → AI 生成代码 → 实时验证 → 对话式修正
  • 核心差异：你不需要先"学会"再"动手"，而是在对话中边探索边实现
实际工作流
  • 需求即 Prompt：把想要的功能直接告诉 AI，不需要写设计文档
  • 探索即开发：不确定怎么做？让 AI 先探索（读代码、查接口），边探索边实现
  • 测试即反馈：看到效果不对？直接告诉 AI 哪里不对，它立即修正
  • 对话即迭代：整个开发过程就是一段持续的对话
真实案例：添加好友功能
对话 1 "我想让 AI 能自动加 SeaTalk 好友"
AI 响应 先分析 SeaTalk 的 Redux Store，发现 CONTACT_ACTION
对话 2 "加好友会弹窗打断我操作，能后台完成吗？"
AI 响应 探索 Webpack chunk → 找到 ContactService → 实现 RPC
对话 3 "测试一下，给 victor.alves 加好友"
AI 响应 执行 → 成功 → 封装为 MCP 工具

从 0 到功能上线：3 轮对话，~30 分钟
关键心法
• 不要试图一次描述完所有需求，分步推进
• 遇到问题不要自己改代码，把现象告诉 AI
• 善用 "先探索再实现"——让 AI 读源码找方案
• 每次迭代保持小步快走，及时验证
15 — SKILL
Skill 开发
把重复的工作流沉淀为标准化指令，AI 一键执行完整流程
什么是 Skill？
Skill 就是一份 Markdown 文件（SKILL.md），里面定义了 AI 在特定场景下应该按什么步骤、用什么工具、产出什么结果。

当用户描述的问题匹配 Skill 的触发条件时，AI 自动加载并按步骤执行。
开发一个 Skill 的过程
  • 从实际工作中提炼：你反复做的一套操作就是 Skill 的雏形
  • 对话式编排：告诉 AI "帮我写一个排查 Bug 的 Skill"，AI 根据你平时的工作流生成初版
  • 实战打磨：在真实场景中使用，发现步骤缺失或顺序不对就迭代
  • 团队共享：Cursor 时代沉淀在 .cursor/skills/，迁移到 Codex 后在 .agents/skills/ 建 repo-scoped 入口
示例：spx-bug-trace
触发：用户说"这个数据不对"、"帮我查一下这个 Bug"

自动执行：
① 浏览器抓 API 请求 → ② 查接口血缘 → ③ 定位 SQL 和源表 →
④ Presto/CK 直查验证 → ⑤ 输出结论 → ⑥ 沉淀到 Confluence

依赖 MCP：api-trace, presto-query, ck-query, browser, atlassian, google-sheets
Skill 文件结构

└── spx-bug-trace/
    ├── SKILL.md
    ├── tools-reference.md
    └── table-mapping.md

.agents/skills/            ← Codex repo-scoped 入口
└── spx-bug-trace/
    ├── SKILL.md             轻量入口：触发 + 流程骨架
    └── references/          按需加载的规则与知识
16 — MCP DEV
MCP 工具开发
给 AI 装上新的"手"——一个 Python 函数就是一个 AI 工具
MCP 工具的本质
MCP（Model Context Protocol）工具就是一个带有描述信息的函数。AI 看到工具的名称和描述，自动决定什么时候调用、传什么参数。
开发流程
  • 确定能力：AI 需要什么能力？查数据库？读消息？调接口？
  • 写 Python 函数：用 @server.tool() 装饰器注册为 MCP 工具
  • 写好描述：工具名、参数说明、返回格式——这是 AI 理解工具的唯一依据
  • 注册到 MCP 配置：Cursor 时代是 ~/.cursor/mcp.json，迁移后 Codex 走 .codex/config.toml / ~/.codex/config.toml
  • 对话验证：直接在对话中让 AI 调用，看效果

KEY TAKEAWAYS
核心技巧总结
  • CDP 是 Electron 应用的万能钥匙 — 一个调试端口参数就能注入任意代码
  • addBinding + evaluate = 零成本双向 RPC — 不需要 WebSocket server 或 HTTP 接口
  • React Fiber 逆向可以调用内部 API — 不修改源码也能实现发消息、加好友
  • 框架兼容性强 — 公司不让继续用 Cursor 后，我们仍能只替换 Agent 接入层，把整套系统迁到 Codex
  • Alarm Bot 告警排查 — 规则匹配 + Agent 自动排查 + 线程回复；prompt 只写正文，与宿主外层标题对齐
  • Remote 远程控制 — Redux 监听 + 独立 Agent 进程，手机也能给 AI 下指令
  • 所见即所得的 AI 开发 — 描述意图 → AI 实现 → 验证效果 → 对话式迭代
  • Skill 沉淀工作流 — 写一份 Markdown，团队所有人都能一键复用
  • MCP 一个函数一个工具 — 给 AI 加新能力只需写一个 Python 函数
